## 2. 核心使用者流程

### 2.1 零售商首次註冊（LINE 登入）

**目標**: 降低註冊摩擦，30 秒內完成

```
1. 首頁 → 點「我要進貨」按鈕
2. 登入頁 → 點「用 LINE 登入」
3. 跳轉 LINE 授權頁
4. 使用者同意授權
5. 跳回我們的 callback，自動建立帳號
6. 導向「補齊資料頁」（因為 LINE 沒有 email 和地址）
   - 必填：Email、電話、店名、收件地址
7. 完成 → 導向零售商首頁
```

**關鍵頁面**:
- `/login`
- LINE 授權頁（外部）
- `/auth/callback`
- `/retailer/onboarding`（補資料頁）
- `/retailer/home`

### 2.2 零售商瀏覽商品 → 下單

**目標**: 主要業務流程

```
1. 零售商首頁
   - 看到「為您推薦」區塊（AI 推薦）
   - 每個商品卡片顯示 LLM 生成的推薦理由
2. 點擊商品 → 商品詳情頁
   - 查看大圖、尺寸、顏色選項
   - 看「視覺相似商品」（CLIP 推薦）
3. 選擇尺寸/顏色/數量 → 加入購物車
   - 購物車圖示右上角顯示數量
4. 繼續逛其他商品 → 加入更多商品
5. 點購物車圖示 → 購物車頁
   - 可調整數量、刪除項目
   - 顯示小計、運費、總計
6. 點「結帳」→ 結帳頁
   - 確認收件資訊（從 profile 帶入，可修改）
   - 選擇付款方式（v1 先只支援匯款）
   - 填寫備註
7. 點「送出訂單」
8. 成功頁
   - 顯示訂單編號
   - 提示「請於 3 日內匯款」
9. 自動導向訂單詳情頁
```

**關鍵頁面**:
- `/retailer/home`
- `/products/:id`
- `/retailer/cart`
- `/retailer/checkout`
- `/retailer/orders/:id`

### 2.3 零售商以圖搜圖

**目標**: 展示 CLIP 視覺搜尋能力

```
1. 任何頁面 → 搜尋列右側有相機圖示
2. 點相機圖示 → 彈出上傳對話框
3. 上傳一張圖片（例如其他地方看到的可愛童裝）
4. 跳轉 /retailer/search?type=image&queryId=xxx
5. 顯示視覺相似的商品清單（跨所有商城）
6. 可進一步篩選（品類、價格、尺寸）
```

**關鍵頁面**:
- `/retailer/search` (接收 query 參數顯示結果)

### 2.4 零售商以文搜圖

```
1. 搜尋列輸入「藍色卡通動物上衣」→ 按 Enter
2. 系統判斷這是語意搜尋（不是商品名精確比對）
3. 後端呼叫 CLIP 把文字轉向量
4. 搜尋最相似的商品
5. 顯示結果
```

### 2.5 批發商上架商品（AI 輔助）

**目標**: 展示 AI 自動建議標籤

```
1. 批發商儀表板 → 點「新增商品」
2. 上傳主圖（支援拖拉、多張）
3. 【AI 魔法時刻】主圖上傳後，後端自動：
   - 萃取 CLIP 向量
   - 呼叫 /tags/suggest 取得建議標籤
   - 前端自動填入建議標籤（使用者可修改）
4. 填寫商品資訊：
   - 名稱、描述、品類
   - 年齡範圍、性別
   - 建議售價
5. 新增 SKU 變體
   - 尺寸（80cm、90cm、100cm...）
   - 顏色（粉紅、天藍、米白...）
   - 每個組合的庫存
6. 預覽 → 送出
7. 成功後跳到商品詳情頁
```

**關鍵頁面**:
- `/wholesaler/dashboard`
- `/wholesaler/products/new`

### 2.6 批發商處理訂單

```
1. 批發商儀表板 → 看到「待處理訂單」紅點
2. 點進訂單列表 /wholesaler/orders
3. 篩選狀態 = "pending"
4. 點單一訂單 → 訂單詳情
5. 確認匯款到帳後，點「標記為已付款」
   - 狀態變為 paid
6. 備貨完成後，點「開始出貨」
   - 狀態變為 processing
   - 可填寫備註（如物流單號）
7. 出貨後點「標記為已出貨」
   - 狀態變為 shipped
   - 零售商會收到通知（email 或 LINE）
8. 零售商簽收後點「完成」
   - 狀態變為 completed
```

**訂單狀態機**:
```
pending ──→ paid ──→ processing ──→ shipped ──→ completed
   │          │
   ↓          ↓
 cancelled  refunded
```

### 2.7 批發商查看數據分析

```
1. 點「數據分析」
2. 儀表板 顯示：
   - 本月訂單數、營收、平均客單價
   - 訂單狀態分佈（圓餅圖）
3. 熱銷商品排行（Top 10）
4. 零售商偏好分析
   - 點某零售商 → 看他的偏好標籤、購買品類分佈
5. AI 洞察區塊
   - 「您的零售商 A 最近都買上衣，建議推薦褲子」
```

## 4. 響應式設計要求

### 4.1 斷點

- Mobile: < 640px
- Tablet: 640-1024px
- Desktop: > 1024px

### 4.2 Mobile 優先原則

B2B 使用者大量手機操作，所有頁面**Mobile 體驗必須良好**：
- 零售商頁面: Mobile First
- 批發商後台: Tablet/Desktop 優先（複雜表格在手機上難用）

### 4.3 關鍵 Mobile 流程

必須在手機上流暢運作：
- 登入、註冊
- 瀏覽商品、加入購物車
- 結帳
- 查看訂單

批發商後台的複雜操作（如數據分析）可以只支援桌面版。

## 5. 狀態管理

### 5.1 全域狀態（Zustand）

- `useAuthStore` — 使用者資料、token
- `useCartStore` — 購物車內容
- `useToastStore` — 全域提示

### 5.2 伺服器狀態（TanStack Query）

- 所有 API 呼叫都用 useQuery / useMutation
- Query Keys 命名: `['products', { shopId, category }]`
- Mutation 成功後 invalidate 相關 query

## 6. 載入與錯誤狀態

### 6.1 載入狀態

每個頁面都要有：
- **Loading skeleton** — 初次載入（不要用 spinner）
- **Pagination loading** — 翻頁時底部顯示 spinner
- **Optimistic UI** — 加購物車等快速操作不等後端

### 6.2 錯誤狀態

- **404 頁面** — 友善的「找不到」設計
- **500 頁面** — 含「重試」按鈕
- **API 錯誤** — 用 Toast 顯示錯誤訊息
- **表單錯誤** — 欄位下方顯示紅色提示

### 6.3 空狀態

- 無商品: 「這家店還沒有上架商品」
- 無訂單: 「還沒有訂單，去逛逛吧！」
- 無推薦: 「繼續瀏覽幾件商品，我們就能推薦您喜歡的款式」

## 7. 通知機制

### 7.1 站內通知

- 訂單狀態變更時通知零售商
- 新訂單時通知批發商

### 7.2 外部通知（v2 功能）

- Email: 註冊驗證、密碼重設、訂單確認
- LINE Notify: 訂單狀態變更（需使用者授權）
